# 8 卡 A100 上的 EffOPD 下一步计划

## 前提判断

当前 EffOPD 计划硬约束是 **non-colocate + G2 cf_l1oo + SGLang OPD**。因此这台 8 卡 A100 最适合全部给 student：actor / critic / rollout 共享这 8 卡，teacher 走外部 SGLang endpoint。

如果没有外部 teacher，只能走 `local_split` 一机 4+4 调试，但现有 EffOPD 第一版明确禁用 colocate/local_split 语义，所以不建议用它验证 EffOPD。可以先用 local_split 验证普通 G2+OPD，不作为 EffOPD 机制验证结果。

## Step 1: 等环境构建结束

当前终端还在安装依赖。先等 `build_conda.sh` / pip 安装完成，不要同时启动 Ray 或训练。

完成后确认：

- `source /root/slime_runtime/slime_env.sh`
- `which python` 指向 `/root/venvs/slime/bin/python`
- `python -V` 正常
- `ray --version` 正常
- `nvidia-smi` 看到 8 张 A100 空闲或接近空闲

## Step 2: 代码健康检查

先只做轻量检查，避免直接烧 8 卡：

```bash
cd /mnt/data/distribution-matching-slime/code/slime-0.2.4
python -m compileall \

python - <<'PY'
```

```bash
python -m pytest tests/test_effopd_core.py
```

## Step 3: 准备 teacher endpoint

推荐使用外部 teacher：

- `TEACHER_API_BASE=http://<teacher-host>:30000`
- `OPD_TEACHER_RM_URL=${TEACHER_API_BASE}/generate`
- `TEACHER_MODEL_NAME=qwen3.5-27b`
- `TEACHER_API_STYLE=sglang_generate`

先用脚本自带 preflight 验证 teacher。不要先跑训练。

如果没有外部 teacher，下一步应先找第二台机器或启动远端 teacher；不要用 8 卡单机 local_split 来验证 EffOPD。

## Step 4: 跑最小 EffOPD shadow smoke

目标：只验证 trigger、sidecar、delta apply/revert、日志和强制 sync，不真实接受外推。

建议命令：

```bash
TEACHER_API_BASE=http://<teacher-host>:30000 \
TEACHER_MODEL_NAME=qwen3.5-27b \
OPD_TEACHER_RM_URL=http://<teacher-host>:30000/generate \
ENABLE_EFFOPD=true \
EFFOPD_VALIDATION_MODE=opd_kl_shadow_cf \
EFFOPD_MAX_TRIGGERS=3 \
NUM_ROLLOUT=4 \
SAVE_INTERVAL=1 \
ENABLE_ASYNC_TRAIN=true \
COLOCATE=false \
bash exper_scripts/main_test/run_g2_opd_qwen35_2b_effopd_smoke.sh
```

预期：

- step 1/2/4 有 EffOPD trigger 日志。
- 日志里分开出现 `cf_l1oo_reward_mean` 和 `opd_reverse_kl_mean`。
- `effopd_state_*.json` 和 `prev_power_*.pt` 出现在 `${SAVE_PATH}/effopd/`。
- shadow 模式下不应真实改变训练目标，主要验证机制 plumbing。

## Step 5: 检查 smoke 结果

重点看：

- Ray job 是否成功结束。
- `argv.sh` 里是否有 `--use-effopd`。
- driver log 中是否有 `EffOPD enabled for G2 cf_l1oo + SGLang OPD`。
- 是否出现 `EffOPD result at rollout_id=0/1/3`。
- SGLang `weight_version` 在 EffOPD trigger 后是否更新。
- checkpoint 是否是 post-save，即保存发生在 EffOPD hook 后。

## Step 6: 升级到 combined shadow

如果 Step 4 正常，再跑：

```bash
EFFOPD_VALIDATION_MODE=combined_shadow \
EFFOPD_MAX_TRIGGERS=3 \
NUM_ROLLOUT=4 \
SAVE_INTERVAL=1 \
...同上...
```

目标：比较 OPD KL 与 cf_l1oo reward 是否同向，不做真实 gate。

判断：

- 如果 `cf_l1oo_reward_mean` 和 `opd_reverse_kl_mean` 在 candidate/trigger 上趋势不严重冲突，才考虑下一步。
- 如果冲突明显，先不要开 `combined_gate`，需要回到 validation 设计。

## Step 7: 小心打开 combined_gate

只在前两步稳定后，做一次极小 gate：

```bash
EFFOPD_VALIDATION_MODE=combined_gate \
EFFOPD_MAX_TRIGGERS=1 \
NUM_ROLLOUT=2 \
SAVE_INTERVAL=1 \
...同上...
```

目标：验证真实接受外推后：

- actor 权重变化。
- lr 按 `EFFOPD_LR_DECAY=0.5` 缩小。
- 下一轮 rollout 使用外推后权重。
- resume sidecar 能正确写出。

## Step 8: Resume 验证

用 Step 7 产出的 checkpoint 做 resume：

- 保持同一个 `RUN_NAME` / `LOAD_PATH`。
- 设置 `NUM_ROLLOUT` 比上次更大。
- 确认加载 `effopd_state_latest.json`。
- 下一次 power-of-two step 能继续触发，不从 0 重新开始。

## Step 9: 再考虑长跑

只有 Step 4-8 全部通过后，再跑稍长：

```bash
EFFOPD_VALIDATION_MODE=combined_shadow \
EFFOPD_MAX_TRIGGERS=4 \
NUM_ROLLOUT=8 \
SAVE_INTERVAL=1 \
...同上...
```

这一步仍以机制验证为主，不急着 claim 3x speedup。

## 不建议现在做

- 不建议 8 卡单机 `local_split` 验证 EffOPD，因为 teacher+student 4+4 会触发 colocate/资源压缩语义，和当前 EffOPD 约束不一致。
- 不建议直接跑 `combined_gate` 长训练。
- 不建议先开 post-eval / MBPP / HumanEval，等机制 smoke 通过后再做。
- 不建议在 pip/conda 安装还未完成时启动 Ray。
